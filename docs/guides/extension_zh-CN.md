# 扩展开发指南

LinkWork 中每个 AI 员工的能力并非只由 Prompt 决定。从容器环境到运行时工具，AI 员工的能力体系由多个层次共同构成。本指南介绍每个层次的扩展方式。

---

## 能力体系总览

| 层次 | 说明 | 扩展方式 |
|------|------|---------|
| **岗位 (Role)** | AI 员工的完整定义模板 | 管理界面创建，配置职责、Prompt、资源配额 |
| **Skills** | 可装卸的知识与流程模块 | 编写 SKILL.md，上传到 Skills 市场 |
| **MCP 工具** | 标准化外部能力接入 | 注册 MCP 服务端点，授权到岗位 |
| **文件管理** | 用户文件空间与岗位工作空间 | 平台自动挂载，按用户/岗位隔离 |
| **Git 项目** | 代码仓库的克隆、操作与交付 | 岗位配置 Git 仓库，运行时自动注入凭证 |
| **容器环境** | 岗位镜像中预装的工具链和运行时 | 构建时固化（Python、Node.js、Java、Go 等） |
| **CLI 工具** | 用户自行安装的命令行工具 | *规划中 — 未来支持用户按需扩展岗位工具链* |

---

## 创建自定义岗位

岗位（Role）是 AI 员工的完整定义，包含职责、能力和约束。

### 岗位配置要素

| 要素 | 说明 | 示例 |
|------|------|------|
| 名称 | 岗位标识 | 「前端开发工程师」 |
| 职责描述 | AI 的角色定位和工作范围 | "负责前端代码审查和 Bug 修复" |
| Prompt | 岗位级别的系统提示词 | 包含技术栈偏好、输出规范等 |
| Skills | 配备的技能模块 | code-review, git-workflow |
| MCP 工具 | 授权的外部工具 | Tavily 搜索、GitLab、数据库 |
| 安全策略 | 命令执行边界 | 允许 git 操作，禁止 rm -rf |
| 资源配额 | CPU、内存限制 | 1 CPU, 2Gi 内存 |
| 模型 | 使用的 LLM | Claude, Qwen, DeepSeek |

### 创建流程

1. **在管理界面创建岗位**，填写名称和职责描述
2. **编写岗位 Prompt**，明确 AI 的角色定位、工作风格和输出规范
3. **选择 Skills**，从市场选择或创建自定义 Skills
4. **配置 MCP 工具**，授权岗位可使用的外部工具
5. **设定安全策略**，定义允许和禁止的操作
6. **构建镜像**，Skills、MCP、策略自动固化到镜像
7. **启动并验证**，下发测试任务验证岗位能力

### 岗位设计最佳实践

| 实践 | 说明 |
|------|------|
| 职责单一 | 每个岗位聚焦一类工作，不要大而全 |
| Prompt 精炼 | 明确角色边界，避免模糊描述 |
| 最小权限 | 只授权必要的 Skills 和工具 |
| 迭代验证 | 先小范围测试，再推广使用 |

## 编写自定义 Skill

Skill 是 AI 员工的可装卸能力模块，本质上是一组结构化的知识文件。构建岗位镜像时，Skills 被注入到 `/opt/agent/skills/` 目录，任务执行时同步到工作目录加载。

### Skill 目录结构

```
my-skill/
├── SKILL.md           # Skill 定义文件（必需）
├── guidelines.md      # 知识文件
├── checklist.md       # 知识文件
└── templates/         # 模板文件（可选）
    └── report.md
```

### SKILL.md 格式

`SKILL.md` 是 Skill 的入口定义文件，包含元数据和核心知识：

```markdown
---
name: code-review
version: 1.0.0
description: 代码审查技能，提供代码质量、安全性和性能审查能力
tags: [development, review]
---

# 代码审查 Skill

## 审查原则

（你的审查知识和指南）

## 审查清单

（具体的检查项）
```

### 元数据字段

| 字段 | 必需 | 说明 |
|------|------|------|
| name | 是 | Skill 唯一标识，建议用 kebab-case |
| version | 是 | 语义化版本号（SemVer） |
| description | 是 | 简要描述 Skill 功能 |
| tags | 否 | 标签，便于市场检索 |

### 知识注入机制

Skill 中的 Markdown 文件会在任务执行时自动加载到 AI 的上下文中。编写 Skill 知识时注意：

| 原则 | 说明 |
|------|------|
| 结构化 | 使用标题、列表、表格组织知识 |
| 可操作 | 提供具体的操作步骤和检查清单 |
| 有边界 | 明确 Skill 的适用范围和限制 |
| 适量 | 控制知识量，避免过多内容稀释注意力 |

Skill 还可以声明依赖（Python/Node/Go 包），构建时自动检查并安装。如果 Skill 依赖某个 MCP 工具（如 Tavily 搜索），需要确保岗位已绑定对应的 MCP 服务。

### 发布到 Skills 市场

1. 在本地开发和验证 Skill
2. 通过管理界面上传 Skill 文件
3. 填写版本说明和变更记录
4. 设置可见范围（个人 / 团队 / 全局）
5. 提交审核（如果是全局发布）

## 注册 MCP 工具

MCP（Model Context Protocol）工具为 AI 员工接入外部能力。

### 支持的工具类型

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| HTTP | 标准 HTTP API 服务 | REST API、数据库查询、第三方服务 |
| SSE | Server-Sent Events 服务 | 流式数据源、实时监控 |

### 注册流程

1. **进入 MCP 管理页面**
2. **填写工具信息**：
   - 名称和描述
   - 服务类型（HTTP / SSE）
   - 服务地址
   - 鉴权配置（如 API Key）
   - 健康检查地址（可选）
3. **保存并等待健康检查通过**
4. **在岗位配置中关联该工具**

### 工具鉴权

MCP 工具的鉴权凭证由平台统一管理：

- 凭证在注册时配置，存储在平台侧
- AI 员工调用工具时，网关自动注入鉴权信息
- AI 员工本身不持有任何外部凭证

### 健康检查

注册后，平台会自动进行定时健康检查：

- 在线：服务可用，响应正常
- 降级：响应延迟较高或间歇性失败
- 离线：连续多次探活失败

岗位配置页面会展示每个工具的实时状态，帮助管理员在下发任务前确认工具可用性。

## 文件管理

每个 AI 员工拥有结构化的文件空间，由平台在任务启动时自动挂载（`fs_prepare`），任务结束后自动清理（`fs_cleanup`）。

| 路径 | 用途 | 说明 |
|------|------|------|
| `/workspace/current/{taskId}` | 当前任务工作目录 | 任务执行的主目录 |
| `/workspace/doc/user` | 用户文件空间 | 挂载到 NFS，按用户隔离，跨任务持久 |
| `/workspace/doc/job` | 岗位工作空间 | 挂载到 NFS，按岗位隔离，跨任务持久 |
| `/workspace/logs` | 任务日志 | 结构化日志输出 |

用户文件空间和岗位工作空间通过 NFS 挂载实现跨任务持久化，AI 员工可以在不同任务间积累和复用知识。

## Git 项目

AI 员工支持在真实代码仓库中工作，Git 集成分为两个阶段：

### 构建时

岗位配置中指定需要预 clone 的 Git 仓库，构建镜像时自动拉取代码到工作目录。

### 运行时

任务启动后，平台自动注入 Git 凭证（GitLab PAT / OAuth Token）：

- `git fetch`、`git pull`、`git push`、`git clone` 等操作自动携带凭证
- `git commit` 自动注入用户身份信息
- 凭证仅在内存中存在，不落盘

任务完成后支持自动 commit/push 并创建 Merge Request，产出直接进入标准 Code Review 流程。

## 容器环境

每个岗位镜像基于 Rocky Linux 9 构建，预装以下工具链：

| 类别 | 工具 |
|------|------|
| 语言运行时 | Python 3.12、Node.js 24、Java 21、Go 1.22 |
| 包管理 | uv/uvx (Python)、npm (Node.js) |
| 版本控制 | git |
| 网络工具 | curl、wget、jq |
| AI 工具 | Claude CLI、@anthropic-ai/claude-code |

岗位的 Skills 还可以声明额外依赖，在构建时自动安装到镜像中。所有工具在构建时固化，运行时只读，保证环境一致性。

---

## CLI 工具（规划中）

当前岗位的工具链在构建时固定，用户无法在运行时自行安装额外的 CLI 工具。未来计划支持：

- 用户在岗位配置中声明需要额外安装的 CLI 工具
- 构建时自动安装并固化到镜像
- 支持自定义安装脚本

## 三层 Prompt 策略

LinkWork 的 Prompt 由三层构成，共同定义 AI 员工的行为：

| 层级 | 来源 | 说明 |
|------|------|------|
| 平台 Prompt | 平台内置 | 通用安全规则、输出格式规范 |
| 岗位 Prompt | 岗位配置 | 角色定位、技术栈偏好、工作风格 |
| 用户 Soul | 用户配置 | 个人偏好、沟通风格、特殊要求 |

**平台 Prompt + 岗位 Prompt + 用户 Soul → 完整的任务上下文。**

### 岗位 Prompt 编写建议

```markdown
# 你是 [岗位名称]

## 角色定位
（简要描述角色和职责）

## 工作风格
（输出风格、详细程度、语言偏好等）

## 技术栈
（相关的技术栈和工具偏好）

## 输出规范
（交付物格式、命名规范等）
```
