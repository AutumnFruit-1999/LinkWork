# 岗位模型

LinkWork 的核心管理单元是**岗位（Workstation）**。一个岗位定义了一个 AI 员工的完整能力画像 — 包括职责、技能、工具权限和安全策略。

---

## 核心思想

传统的 AI Agent 是一个进程，跑在宿主机上，没有边界、没有约束。LinkWork 的做法截然不同：

**每个 AI 员工都是一个容器化服务。**

每个 AI 员工在独立的 Docker / K8s 容器中运行，拥有完全隔离的执行环境。像管理微服务集群一样管理 AI 团队，全面复用云原生生态。

---

## 三层模型

```
Workstation（岗位）
  └── Instance（实例）
        └── Task（任务）
```

### Workstation — 岗位定义

岗位是一个**不可变的能力模板**，定义了 AI 员工「是谁」和「能做什么」：

| 属性 | 说明 |
|------|------|
| 名称与职责 | 岗位的角色定位和职责描述 |
| Skills 列表 | 该岗位配备的技能模块 |
| MCP 工具 | 该岗位可使用的外部工具 |
| 安全策略 | 命令执行的允许/禁止/审批规则 |
| 资源配额 | CPU、内存、磁盘配额 |
| 模型配置 | 使用的 LLM 模型及参数 |

> 创建一个「前端开发工程师」岗位，任何 AI 模型实例化后都能直接上岗。换模型不换岗，换人不换能力。

### Instance — 岗位实例

一个岗位可以同时运行多个实例，每个实例是一个独立的容器：

- 实例之间完全隔离（文件系统、网络、进程）
- 每个实例有独立的工作空间和执行上下文
- 实例数量可以按需扩缩容

### Task — 任务

任务是用户下发给 AI 员工的具体工作指令：

- 每个任务在一个实例中执行
- 任务有完整的生命周期：排队 → 执行中 → 完成/失败
- 执行过程全程可观测，支持 WebSocket 实时追踪

---

## 容器化运行

AI 员工的容器化运行带来以下保障：

### 隔离性

- **文件系统隔离** — 每个员工有独立的文件系统，互不可见
- **网络隔离** — 默认关闭外部网络，仅按需放行必要服务
- **进程隔离** — AI 推理进程与命令执行进程分离运行

### 资源治理

- **按岗位设定配额** — CPU、内存按需分配，防止单个员工拖垮集群
- **弹性伸缩** — 按任务量自动扩缩容，空闲时自动释放资源
- **故障自愈** — 容器崩溃自动重启，残留工作目录自动清理

### 持久化

- **工作空间持久化** — 任务产出和中间状态跨会话保留
- **长期记忆** — 知识沉淀和向量记忆跨任务持续积累

---

## 岗位生命周期

```
创建岗位 → 配置 Skills/MCP/策略 → 构建镜像 → 启动实例 → 接收任务 → 执行 → 空闲回收
```

1. **创建**：管理员定义岗位名称、职责、能力配置
2. **构建**：调度引擎自动构建岗位镜像（Skills 注入、MCP 固化、策略内嵌）
3. **启动**：在 K8s 集群中创建容器实例
4. **运行**：实例从任务队列中消费任务并执行
5. **回收**：空闲超时后自动释放资源

> 配置变更必须重新构建镜像。这是刻意的设计选择 — 详见 [Harness Engineering](./harness-engineering_zh-CN.md)。

---

## 与传统 Agent 的区别

| 维度 | 传统 Agent | LinkWork 岗位 |
|------|-----------|--------------|
| 运行方式 | 宿主机进程 | 独立容器 |
| 能力管理 | 运行时随意安装 | 构建时锁定，管理员审批 |
| 隔离性 | 无 | 文件/网络/进程完全隔离 |
| 资源控制 | 无 | K8s 原生资源配额 |
| 复用性 | 个人配置，难以复制 | 岗位模板，一键复制 |
| 安全边界 | 依赖用户自觉 | 策略引擎 + 审批流 |
