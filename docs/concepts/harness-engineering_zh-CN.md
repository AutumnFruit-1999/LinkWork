# Harness Engineering

AI Agent 的成功率不只取决于模型能力 — **执行环境的确定性同样关键**。

LinkWork 采用 **「一岗位一镜像」** 范式：Skills、MCP 工具、安全策略全部在构建时固化到容器镜像，运行时只读不写，彻底杜绝环境漂移。

---

## 为什么需要 Harness Engineering

传统 Agent 框架通常在运行时动态安装插件、拉取配置、加载工具。这带来一系列问题：

| 问题 | 说明 |
|------|------|
| 环境漂移 | 不同时间启动的实例，配置可能不一致 |
| 不可复现 | 昨天能跑的今天可能报错，无法定位原因 |
| 版本混乱 | 哪个实例用了哪个版本的插件，无从追溯 |
| 安全风险 | AI 可能在运行时自行安装不受控的工具 |

Harness Engineering 的核心理念是：**把所有不确定性消灭在构建阶段**。

---

## 构建时固化

每次岗位构建，调度引擎自动执行完整的装配流程：

### 1. Skills 注入

按岗位配置拉取对应版本的 Skills，锁定精确版本号写入镜像。

```
岗位配置: skills = [code-review@1.2.0, git-workflow@2.0.1]
    ↓
构建引擎拉取 Skills 文件
    ↓
写入镜像: /opt/agent/skills/code-review/  (v1.2.0)
          /opt/agent/skills/git-workflow/  (v2.0.1)
```

### 2. MCP 配置固化

根据岗位关联的 MCP 工具，生成 MCP 配置描述文件，只读写入镜像。

### 3. 安全策略内嵌

安全策略文件打包进镜像，启动时自动加载。策略定义了该岗位 AI 员工的命令执行边界。

### 4. 版本快照记录

记录每个 Skill 和 MCP 工具的精确版本，构建完全可复现。

---

## 运行时只读

镜像构建完成后，运行时的能力配置是**只读**的：

- AI 员工无法自行安装新的 Skills
- AI 员工无法修改 MCP 工具配置
- AI 员工无法绕过安全策略

**配置变更 → 必须重新构建镜像。**

这是刻意的设计选择：保证每个运行中的 AI 员工环境完全可预测、可复现。

---

## 上下文注入 (Context Priming)

镜像固化解决了「能力」的确定性，上下文注入解决了「任务启动」的确定性。

任务启动时，运行时自动完成执行上下文装配：

| 步骤 | 说明 |
|------|------|
| Skills 同步 | 将镜像中预装的 Skills 同步到工作目录，按标准路径加载 |
| Git 仓库准备 | 按任务配置自动拉取代码到工作分支 |
| 三层 Prompt 策略 | 平台 Prompt + 岗位 Prompt + 用户 Soul，构建完整的任务背景 |

AI 员工不是从零开始理解任务，而是**带着完整环境和上下文上岗**。

---

## Fail-fast 构建策略

构建阶段采用严格的 Fail-fast 策略：

- 配置了 Skills 但 Git clone 失败 → **中断构建**
- 配置了 MCP 但生成配置失败 → **中断构建**
- 安全策略文件格式错误 → **中断构建**

**绝不静默跳过。** 问题在构建阶段就暴露出来，而不是让 AI 员工带着残缺能力运行。

---

## 总结

> 一岗位一镜像：环境即代码、版本可锁定、构建可复现、问题早暴露。

| 原则 | 说明 |
|------|------|
| 环境即代码 | 岗位的能力配置完全由构建定义，与代码一样可版本管理 |
| 版本可锁定 | 每个 Skill、MCP 工具都锁定精确版本 |
| 构建可复现 | 相同的配置输入，必然产生相同的镜像 |
| 问题早暴露 | 构建阶段 Fail-fast，不让问题延迟到运行时 |
